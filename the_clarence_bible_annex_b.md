1. Executive Summary

The Integrity Engine is a four-layer quality system that ensures every AI-generated output CLARENCE produces — clause explanations, position recommendations, party-specific analysis, negotiation guidance, and training scenarios — becomes more accurate over time through a combination of structured validation, human feedback capture, a growing corrections library, and automated pattern analysis.

This is not model fine-tuning. CLARENCE uses Anthropic's Claude API, which cannot be fine-tuned. Instead, the Integrity Engine builds an ever-growing proprietary context library — validated corrections, error patterns, and domain-specific rules — that is injected into every AI interaction, making each output more accurate than the last.

The result is a system that earns the name The Honest Broker not just through its design philosophy but through a measurable, auditable commitment to output accuracy. The Integrity Engine is a continuous improvement mechanism that compounds in value with every interaction.


2. The Problem

CLARENCE generates thousands of AI outputs per day across its modules: clause explanations in the Contract Studio, position recommendations during mediation, party-specific analysis in strategic assessments, opponent behaviour in Training Studio sessions, and compliance guidance from the Risk Mitigation Engine.

These outputs are generated by Claude (Anthropic's large language model) using carefully constructed prompts enriched with session context — party data, clause content, deal profile, leverage calculations, playbook rules, and negotiation history. The quality of these outputs is generally high, but three recurring issues have been identified through beta testing:

2.1 Party Name Confusion. CLARENCE occasionally uses incorrect party names — referring to "the customer" when it should say "Ventrica," or swapping which party is the disclosing party and which is the receiving party in an NDA context. This undermines trust immediately, even when the underlying analysis is correct.

2.2 Terminology Imprecision. Legal descriptions sometimes use terminology that is technically imprecise for the specific contract type. An NDA explanation might reference "service levels" (a concept from outsourcing agreements, not NDAs). A position recommendation might describe a clause's effect in language more suited to a different jurisdiction or contract category.

2.3 Position-Data Misalignment. AI-generated text occasionally references positions or recommendations that do not match the actual numeric data in the session. CLARENCE might state "your position of 7" when the stored position is 4, or describe a "narrow gap" when the data shows a gap of 6.

Each of these issues has been addressed individually through prompt engineering — adjusting the system prompts, adding validation instructions, enriching the context. These fixes work for the specific case but do not prevent similar errors in new contexts. There is no system that learns from corrections and applies them proactively.

2.4 The Limitation of Self-Correction. Research into LLM self-correction (MIT Press, 2024; multiple studies through 2025-2026) demonstrates a critical finding: asking a language model to review and correct its own outputs without external grounding actually degrades accuracy. Models exhibit self-bias, changing correct answers to wrong ones as frequently as they fix genuine errors. Pure self-correction — "read your output and fix any mistakes" — is not a viable strategy.

The Integrity Engine addresses this by grounding every correction in either hard data (structured validation against session facts) or human expertise (feedback from users and legal partners). The AI is never asked to judge its own work in a vacuum.


3. Architecture Overview

The Integrity Engine operates as four complementary layers, each contributing a different type of quality assurance:

    Layer 1: Structured Quality Gate       → Real-time, every output
    Layer 2: Corrections Library            → Retrieval-augmented context
    Layer 3: Human Feedback Capture         → Multi-tier signal collection
    Layer 4: Batch Pattern Analysis         → Periodic autonomous review

Each layer feeds into the others. The Quality Gate catches errors and generates correction candidates. Human feedback validates those candidates and surfaces new ones. The Corrections Library stores validated corrections for injection into future prompts. Batch Analysis identifies systemic patterns that individual interactions miss.

The compound effect: Month 1 might have 20 correction rules. Month 6 might have 500. Month 12 might have 2,000. Each one makes CLARENCE more accurate for its specific context, and the library never forgets.


4. Layer 1: Structured Quality Gate

4.1 Purpose

A real-time validation pass that checks every CLARENCE output against known facts from the session data before the output reaches the user. This is deterministic checking — not AI judgement — which means it is fast, cheap, and reliable.

4.2 What It Checks

Party Name Accuracy. The output is scanned for party references and checked against the actual party names in the session (session.customerCompanyName, session.providerCompanyName) and the contract-type-specific role labels from the Party Role Matrix (e.g., discloser/recipient for NDAs, licensor/licensee for IP agreements). If CLARENCE refers to "the provider" when the session's provider is "Ventrica," the system checks whether the context warrants the specific name or the generic label, and flags inconsistencies.

Position Data Accuracy. Any numeric position references in the output (e.g., "your position of 7") are checked against the actual stored positions in the clause data. If the numbers do not match, the output is flagged for correction before display.

Contract Type Terminology. The output is checked against a contract-type-specific terminology allowlist. An NDA context should not reference "service levels," "SLAs," or "performance penalties." An outsourcing agreement should not reference "disclosure obligations" in the NDA sense. Flagged terms are highlighted for correction.

Clause Reference Accuracy. When the output references other clauses by name or number, those references are checked against the actual clause list in the session. Non-existent clause references are flagged.

Internal Consistency. The output is checked for internal contradictions — e.g., describing a position as "strong" while also noting a gap of 8 (which indicates weakness), or recommending acceptance while flagging the clause as a deal breaker.

4.3 Implementation Approach

The Quality Gate operates as a middleware function that processes every AI response before it is returned to the frontend. It receives the raw AI output and the session context, performs the checks described above, and either passes the output through (if clean) or applies corrections and logs the intervention.

For party name and position data checks, the validation is purely algorithmic — string matching and numeric comparison against session data. No additional AI calls are required.

For terminology and consistency checks, a lightweight AI validation call may be used, but critically, this validation call is grounded in the session data and a terminology ruleset — it is not asking the AI to judge its own work in a vacuum.

4.4 Quality Gate Metadata

Every output that passes through the Quality Gate is tagged with metadata:

- quality_score: 0-100 based on the number and severity of issues detected
- corrections_applied: array of corrections made before display
- flags_raised: array of potential issues that were detected but not auto-corrected
- gate_version: version of the Quality Gate rules applied

This metadata is stored alongside the output, enabling Layer 4 (Batch Analysis) to track quality trends over time.


5. Layer 2: Corrections Library

5.1 Purpose

A persistent, growing database of validated corrections that are injected into AI prompts as contextual guidance. This is the system's memory — every correction that has been validated (by the Quality Gate, by human feedback, or by batch analysis) becomes a rule that prevents the same error from recurring.

5.2 Data Model

The Corrections Library is stored in a Supabase table (clarence_corrections) with the following structure:

    correction_id       UUID PRIMARY KEY
    contract_type       TEXT            -- e.g., 'mutual_nda', 'alliance_agreement'
    clause_category     TEXT            -- e.g., 'liability', 'termination', 'confidentiality'
    clause_name         TEXT            -- specific clause name, if applicable
    error_type          TEXT            -- 'party_name', 'terminology', 'position_data', 'legal_accuracy', 'tone', 'other'
    error_pattern       TEXT            -- description of what CLARENCE got wrong
    correction          TEXT            -- what it should say instead
    example_wrong       TEXT            -- an example of the incorrect output
    example_correct     TEXT            -- the corrected version
    source              TEXT            -- 'quality_gate', 'human_feedback', 'batch_analysis', 'expert_review'
    confidence          FLOAT           -- 0.0 to 1.0, how reliable this correction is
    times_applied       INTEGER         -- how often this correction has been injected
    times_effective     INTEGER         -- how often the correction prevented a recurrence
    is_active           BOOLEAN         -- whether this correction is currently in use
    created_at          TIMESTAMPTZ
    updated_at          TIMESTAMPTZ
    created_by          UUID            -- user who validated, if human-sourced

5.3 How Corrections Are Injected

When CLARENCE generates a response, the system queries the Corrections Library for relevant rules based on the current context — contract type, clause category, and clause name. Matching corrections are injected into the system prompt as structured guidance:

    "Previous corrections for this context:
    - For Mutual NDA / Confidentiality clauses: The 'disclosing party' is the party sharing
      confidential information. Do not refer to the disclosing party as the 'provider' —
      in an NDA, both parties may disclose. Use the actual party names from the session.
    - For Mutual NDA contexts: Do not reference 'service levels', 'SLAs', or 'performance
      penalties' — these concepts belong to outsourcing or service agreements, not NDAs."

The number of corrections injected is managed to avoid prompt bloat — typically the 5-10 most relevant corrections for the specific context, prioritised by confidence score and times_effective.

5.4 Correction Lifecycle

New corrections enter the library with a confidence score based on their source:
- Quality Gate detections: 0.6 initial confidence
- Human feedback (single report): 0.7 initial confidence
- Expert review (legal partner): 0.9 initial confidence
- Batch analysis (pattern across multiple sessions): 0.8 initial confidence

Confidence increases when the correction is applied and subsequent outputs for the same context receive positive feedback. Confidence decreases if the correction is applied but the output is still flagged. Corrections with confidence below 0.3 are automatically deactivated for review.


6. Layer 3: Human Feedback Capture

6.1 Purpose

Collect signals from users about output quality — from quick implicit signals to detailed expert reports. Human feedback is the highest-quality training signal because it comes from people who understand the domain and can identify errors that automated checks miss.

6.2 Three Tiers of Feedback

Tier 1: Inline Response Rating (one-click). Every CLARENCE chat message in the Contract Studio, QC Studio, and Training Studio includes a subtle thumbs-up/thumbs-down indicator. This captures volume signal — if 80% of responses about NDA confidentiality clauses receive thumbs-down, there is a systemic issue even if no one writes a detailed report. Implementation: small icons below each AI message, storing clause_id, session_id, response_text, and rating.

Tier 2: Implicit Correction Signal (automatic). When a user takes an action that contradicts a CLARENCE recommendation, the system captures the delta:
- CLARENCE recommends position 7, user sets position 4 → delta captured
- CLARENCE uses party name "the provider," user's next chat message uses "Ventrica" → name preference captured
- CLARENCE explains a clause, user immediately asks the same question differently → comprehension issue captured
- User modifies AI-generated draft text → before/after diff captured

These signals require no user effort — the system observes behaviour and infers feedback.

Tier 3: Detailed Clause Feedback (existing). The ClauseFeedbackButton already deployed in the Contract Studio captures full-context clause issue reports: issue type (wrong position, wrong party, inaccurate analysis, missing context), detailed explanation, and a comprehensive clause snapshot (positions, weights, draft content, recent chat history, session context). This is the richest signal source but lowest volume.

6.3 Feedback Pipeline

All three tiers flow into the same pipeline:

    User interaction → Feedback captured → Stored in feedback tables
                                               ↓
                                    Correction candidate generated
                                               ↓
                              Validated (by volume, expert, or pattern)
                                               ↓
                              Added to Corrections Library (Layer 2)


7. Layer 4: Batch Pattern Analysis

7.1 Purpose

A scheduled process that reviews accumulated outputs and feedback to identify systemic patterns that individual interactions miss. This is the "overnight learning" component — inspired by the autonomous experimentation principle demonstrated in Karpathy's autoresearch project, where an AI agent runs hundreds of experiments overnight, each one building on the last.

7.2 What It Analyses

Error Clustering. Groups Quality Gate interventions by contract type, clause category, and error type to identify systematic weaknesses. Example finding: "CLARENCE swaps party names in 34% of Mutual NDA confidentiality clause explanations but only 3% of outsourcing liability clauses."

Feedback Correlation. Correlates inline ratings (Tier 1) with implicit correction signals (Tier 2) to validate whether low-rated outputs are actually being contradicted by user behaviour, or whether the rating reflects preference rather than accuracy.

Correction Effectiveness. Measures whether corrections in the library are actually preventing recurrence. A correction that has been applied 50 times but the same error keeps appearing (times_effective is low) may indicate the correction is poorly worded or the root cause is deeper than the correction addresses.

Prompt Drift Detection. Monitors whether output quality changes after system updates, new templates, or prompt modifications. If quality scores drop after a deployment, the batch analysis flags it.

7.3 Outputs

The batch analysis produces three outputs:

New Correction Rules. Automatically generated correction candidates based on detected patterns, added to the Corrections Library with source='batch_analysis' and initial confidence based on the strength of the pattern.

Quality Report. A summary dashboard showing accuracy trends over time, most common error types, which contract types and clause categories need attention, and the effectiveness of recent corrections.

Escalation Alerts. Notifications to the team when systemic issues are detected that require human intervention — a new contract type that consistently produces poor outputs, a correction that is not working, or a sudden quality drop.

7.4 Schedule

The batch analysis runs on a configurable schedule — initially weekly during beta, moving to daily as volume increases. The analysis is implemented as an n8n workflow or Supabase edge function that queries the output logs, feedback tables, and corrections library, performs the analysis, and writes results back.


8. The Compound Learning Effect

The Integrity Engine's value is not in any single layer — it is in the compounding interaction between them:

    Week 1:     Quality Gate catches 15 party name errors → 15 correction candidates
    Week 2:     Human feedback validates 12 of them → 12 corrections active
    Week 3:     Batch analysis identifies a pattern: "all 12 are in NDA contexts" →
                1 high-confidence meta-correction for NDA party naming
    Week 4:     Quality Gate detects 3 party name errors (down from 15) → system is learning
    Month 3:    NDA party name errors near zero. System has moved on to catching
                terminology issues in alliance agreements.
    Month 6:    500+ active corrections. New contract types added to the platform
                start with corrections from similar types, accelerating accuracy.
    Year 1:     The Corrections Library is a proprietary dataset that no competitor can
                replicate — it encodes thousands of domain-specific rules learned from
                real negotiation contexts.

This is not theoretical. The architecture is proven in production systems (retrieval-augmented generation with feedback loops) and the individual components — structured validation, corrections databases, feedback capture, batch analysis — are each straightforward to implement. The innovation is in their integration into a unified, self-reinforcing system purpose-built for legal contract negotiation.


9. Relationship to Existing CLARENCE Systems

The Integrity Engine does not replace existing quality mechanisms — it integrates with them:

Session Context Builder (IP Asset #4). The existing context builder assembles AI context from six data sources. The Integrity Engine adds a seventh: the Corrections Library. Relevant corrections are injected alongside clause text, party data, and playbook rules.

Clause Feedback Button (FA-26). The existing ClauseFeedbackButton provides Layer 3, Tier 3 feedback. The Integrity Engine adds Tier 1 (inline ratings) and Tier 2 (implicit signals) to create a complete feedback spectrum.

Party Role Matrix (IP Asset #5). The existing role matrix provides the ground truth for party name validation in Layer 1. The Quality Gate uses the matrix to determine correct party labels for each contract type.

Playbook Compliance Scoring (IP Asset #1). The compliance scoring model validates positions against playbook rules. The Integrity Engine validates AI outputs against session facts. Both are quality assurance mechanisms operating at different levels — compliance checks the negotiation, the Integrity Engine checks the AI.


10. Intellectual Property Position

The Integrity Engine represents a distinct, protectable intellectual property asset. The following elements are novel in the contract negotiation domain:

10.1 The Corrections Library Data Model. A structured, growing database of domain-specific AI output corrections, indexed by contract type, clause category, and error type, with confidence scoring and effectiveness tracking. This dataset becomes more valuable with every interaction and cannot be replicated without processing the same volume of real negotiation contexts.

10.2 The Four-Layer Architecture. The integration of structured validation (deterministic checks against session data), retrieval-augmented correction (injecting validated corrections into prompts), multi-tier human feedback (inline ratings, implicit signals, detailed reports), and batch pattern analysis (autonomous error detection) into a unified, self-reinforcing quality system for AI-generated legal outputs.

10.3 The Compound Learning Model. The feedback loop where each layer's outputs feed into the others — Quality Gate detections become correction candidates, human feedback validates them, the Corrections Library prevents recurrence, and batch analysis identifies systemic patterns — creating a system that improves its own improvement rate over time.

No existing legal-tech platform has a comparable system. Contract lifecycle management tools (Ironclad, Juro, DocuSign CLM) do not generate AI mediation outputs to validate. Legal research platforms (LexisNexis, Practical Law) provide static content that does not adapt. AI drafting tools (Harvey, CoCounsel) may have internal quality processes but do not operate in a mediated, dual-party context where output accuracy affects both sides of a negotiation.


11. Implementation Phases

Phase 1: Structured Quality Gate
Build the real-time validation middleware for party name checking, position data verification, and contract-type terminology validation. This addresses the most visible quality issues immediately.
Dependencies: None. Uses existing session data.

Phase 2: Corrections Library
Create the clarence_corrections table and implement the injection mechanism in the Session Context Builder. Seed the library with known corrections from beta testing feedback.
Dependencies: Phase 1 (Quality Gate generates initial correction candidates).

Phase 3: Human Feedback Expansion
Add inline thumbs-up/thumbs-down to CLARENCE chat messages. Implement implicit correction signal capture (position change deltas, party name preferences).
Dependencies: None (can run in parallel with Phase 2).

Phase 4: Batch Pattern Analysis
Build the scheduled analysis workflow (n8n or Supabase edge function). Create the quality dashboard. Implement escalation alerts.
Dependencies: Phases 1-3 (requires accumulated data to analyse).


12. Success Metrics

The Integrity Engine's effectiveness is measurable:

Quality Gate Intervention Rate. Percentage of outputs requiring correction before display. Target: decrease from estimated 15-20% (current) to below 5% within 6 months.

Correction Library Growth. Number of active, validated corrections. Target: 100+ within 3 months, 500+ within 12 months.

Feedback Volume. Percentage of AI interactions receiving explicit or implicit feedback. Target: 10% explicit (Tier 1), 40% implicit (Tier 2) within 3 months.

Recurrence Rate. Percentage of corrected error types that recur after correction is active. Target: below 10% recurrence for high-confidence corrections.

User Trust Score. Derived from Tier 1 ratings — trending upward indicates the system is learning effectively.

14. Open Questions

14.1 Auto-correction vs flagging. Should the Quality Gate auto-correct detected errors before display, or flag them for user awareness? Auto-correction is seamless but invisible. Flagging builds trust ("CLARENCE caught and fixed an issue") but adds friction. Recommendation: auto-correct with a subtle indicator ("Output verified by Integrity Engine").

14.2 Correction sharing across companies. Should corrections validated by one company's usage benefit other companies? This accelerates learning but raises data isolation questions. Recommendation: corrections are generalised (no company-specific data) and shared across the platform, but companies can also have private corrections.

14.3 Quality dashboard access. Should the quality dashboard (showing accuracy trends, correction counts, error patterns) be visible to customers, or internal only? Making it visible reinforces the "Honest Broker validates itself" narrative but exposes imperfections. Recommendation: offer an optional quality transparency view to enterprise customers.

14.4 Correction authoring. Should power users or legal partners be able to add corrections directly to the library, rather than waiting for the automated pipeline? This would accelerate learning for new contract types. Recommendation: yes, via a "Teach CLARENCE" interface for authorised users.

14.5 Training Studio integration. Should the Integrity Engine also validate Training Studio outputs (opponent behaviour, debrief analysis)? The same error types apply — wrong party names, inaccurate terminology. Recommendation: yes, extend the Quality Gate to training outputs in Phase 2 or 3.
